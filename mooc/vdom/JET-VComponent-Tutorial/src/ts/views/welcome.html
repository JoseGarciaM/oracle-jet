Welcome to VComponents!

The 11.0.0 release of JET introduces a new solution for creating JET Custom Components: the VComponent API.

As with JET's Composite Component Architecture, VComponents provide an easy way for developers to author their own Web Components. VComponents differ from composite components in one key way: VComponents leverage virtual DOM rendering and Preact rather than Knockout-based templates to produce their content. As we will see, this has implications for how VComponents are authored and used.

This tutorial provides an introduction to the new Preact-based VComponent API. It is not intended to be a comprehensive guide to every aspect of VComponents. Instead, the goal of this tutorial is to help onboard new VComponent authors. By the end of this tutorial, you should be able to create your own VComponents and incorporate these into your JET-based projects.

One important thing to note before we dive in: while we are very excited about the VComponent API, we consider this to be an incremental addition rather than revolutionary change to the JET architecture. VComponents are designed to work seamlessly with existing JET applications. There is no need to rewrite your existing JET applications in order to start using VComponents. VComponents can happily coexist with other JET components. This tutorial is itself a demonstration of this principle, as it is a plain old JET app with some VComponents sprinkled throughout.

The same goes for your existing composite component-based JET Custom Components. If you are happy building on top of the Composite Component Architecture, there is no reason for you to disrupt your development plans. The Composite Component Architecture is not going away. We feel that both composite components and VComponents are excellent options for component authoring. You are free to pick whichever approach best meets your team's skill set and preferences.

Prerequisites

The VComponent API builds on top of a number of technologies. In addition to a general understanding of virtual DOM, VComponent authors will need to be familiar with the following:

  • Preact: Implementing a VComponent requires direct use Preact APIs. Getting to know Preact is a good place to start.
  • TypeScript: All VComponents are authored in TypeScript. Static types are your friends!
  • JSX: JSX defines a mechanism for embedding markup content inside of ECMAScript. JSX can also be placed inside of TypeScript code as well.
  • ECMAScript modules (and TypeScript modules): VComponents are authored using ECMAScript-style imports/exports.
  • TypeScript decorators: the VComponent API uses decorators to express information and behavior that would otherwise be difficult to express via code.
  • ojet CLI: we will use this command line tool to build our VComponents.
  • AMD: Although VComponents are authored as ECMAScript modules, ojet CLI transpiles these to AMD for runtime usage.

In addition to the above, there are a few tangential subjects that will likely be of interest to VComponent authors:

  • Composite Component Architecture: while it is not a requirement to master composite components, developers that are familiar with composite components will see conceptual (and practical) connections to the VComponent API.
  • JET Packs: we expect that VComponent authors will typically create multiple, related VComponents, and as such will want to package these into JET Packs.
  • Oracle Visual Builder: whether or not your own application is built using Visual Builder, Visual Builder users can import your JET Custom Components (including VComponents) into their applications.
